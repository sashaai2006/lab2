## 2. Пошаговое выполнение работы

### Шаг 1. Сбор данных

В качестве исходных данных было принято решение использовать **синтетическое (вымышленное) расписание** учебного потока для пяти групп физико-математического профиля. Использование синтетических данных позволяет контролируемо заложить в расписание определённые паттерны (например, «окна» у конкретных групп, свободные дни у преподавателей) для последующей проверки их обнаружения предикатами.

**Характеристики потока:**
- 5 учебных групп (`Б24-501` ... `Б24-505`);
- 10 предметов (Математика, Физика, Информатика и т.д.);
- 5 преподавателей (каждый ведёт 2 предмета);
- 10 аудиторий разной вместимости (А-корпус, Б-корпус и т.д.);
- 5 учебных дней (Понедельник — Пятница);
- 4 временных слота в день (09:00, 10:45, 13:00, 14:45).

### Шаг 2. Моделирование (проектирование структуры фактов)

Для представления расписания в базе знаний Prolog необходимо было выбрать структуру, которая однозначно идентифицирует каждое занятие и позволяет легко проверять ограничения.

Был выбран 6-арный предикат:
```prolog
занятие(Группа, Предмет, Преподаватель, День, Время, Аудитория).
```

**Обоснование выбора атрибутов:**
1. **Группа** — необходима для контроля непересечения занятий у студентов (у одной группы не может быть двух пар одновременно).
2. **Предмет** — информационный атрибут, нужен для запросов вида «где проходит математика».
3. **Преподаватель** — необходим для контроля занятости преподавателя (один человек не может быть в двух аудиториях одновременно).
4. **День** и **Время** — временные координаты занятия. Время хранится как атом (например, `'09:00'`), но для вычисления «окон» введена дополнительная сущность `порядок_времени/2`, задающая целочисленный ранг слота.
5. **Аудитория** — пространственная координата, необходима для контроля вместимости (в одной аудитории не может быть двух разных занятий одновременно).

Все аргументы хранятся как атомы (в одинарных кавычках), что соответствует требованиям базового уровня.

### Шаг 3. Кодирование БЗ (ввод фактов)

На основе спроектированной модели была создана база знаний в файле `schedule.pl`. 

Сначала были явно заданы все домены значений:
```prolog
группа('Б24-501').
...
аудитория('А101').
...
порядок_времени('09:00', 1).
```
Явное задание доменов, особенно `аудитория/1`, критически важно. Если аудитория ни разу не используется в расписании, она всё равно должна числиться в домене, иначе предикат поиска свободных аудиторий её не найдёт.

Затем были закодированы 55 фактов `занятие/6`. Факты вводились с учётом **инвариантов расписания**:
1. У одной группы не более одной пары в один временной слот.
2. Один преподаватель не ведёт двух пар одновременно.
3. Одна аудитория не используется дважды в один слот.
4. Все атомы фактов принадлежат своим доменам.

## 3. Базовые запросы и предикаты (Уровень 1)

### 3.1 `свободна_аудитория(+День, +Время, ?Аудитория)`

Истина, если в данный день в данное время в аудитории нет занятия. При незафиксированном третьем аргументе перебирает весь домен `аудитория/1`.

```prolog
свободна_аудитория(День, Время, Аудитория) :-
    аудитория(Аудитория),
    \+ занятие(_, _, _, День, Время, Аудитория).
```
*Декларативный смысл:* Аудитория свободна, если она существует в природе (принадлежит домену) и **не** существует факта занятия в ней в указанное время.

### 3.2 `нагрузка_группы(?Группа, ?КоличествоПар)`

Подсчитывает общее число занятий у группы за учебную неделю.

```prolog
нагрузка_группы(Группа, КоличествоПар) :-
    группа(Группа),
    findall(_, занятие(Группа, _, _, _, _, _), Список),
    length(Список, КоличествоПар).
```
*Декларативный смысл:* Нагрузка — это длина списка всех фактов занятий, где фигурирует данная группа.

### 3.3 `окна_в_расписании(+Группа, +День)`

Истина, если у группы в указанный день есть хотя бы одно **окно** — пустой временной слот между двумя занятиями. Промежутки до первой пары и после последней пары окнами не считаются.

```prolog
окна_в_расписании(Группа, День) :-
    группа(Группа),
    день(День),
    findall(Слот,
        ( занятие(Группа, _, _, День, Время, _),
          порядок_времени(Время, Слот) ),
        Слоты),
    sort(Слоты, ОтсортированныеСлоты),
    есть_пробел(ОтсортированныеСлоты).

есть_пробел([Прев, Тек | _]) :-
    Тек - Прев > 1, !.
есть_пробел([_, Тек | Хвост]) :-
    есть_пробел([Тек | Хвост]).
```
*Алгоритм:* Собираем номера занятых слотов (через биекцию `порядок_времени/2`), сортируем их. Затем рекурсивно проверяем разность между соседними элементами списка. Если разность > 1, значит между парами есть пропуск.

## 4. Валидация и детекция конфликтов (Уровень 2)

Для обеспечения математической корректности модели в `schedule.pl` добавлены предикаты валидации (детекции конфликтов). Они реализованы через поиск контрпримеров.

**Конфликт группы (две пары одновременно):**
```prolog
конфликт_группы(Группа, День, Время, Ауд1, Ауд2) :-
    занятие(Группа, _, _, День, Время, Ауд1),
    занятие(Группа, _, _, День, Время, Ауд2),
    Ауд1 @< Ауд2. % Избегаем дублирования (А,Б) и (Б,А)
```

**Конфликт преподавателя (в двух местах одновременно):**
```prolog
конфликт_преподавателя(Преп, День, Время, Гр1, Гр2) :-
    занятие(Гр1, _, Преп, День, Время, _),
    занятие(Гр2, _, Преп, День, Время, _),
    Гр1 @< Гр2.
```

**Конфликт аудитории (две группы в одной аудитории):**
```prolog
конфликт_аудитории(Ауд, День, Время, Гр1, Гр2) :-
    занятие(Гр1, _, _, День, Время, Ауд),
    занятие(Гр2, _, _, День, Время, Ауд),
    Гр1 @< Гр2.
```

Комплексная проверка базы `проверить_базу/0` возвращает истину только если ни один из конфликтов не найден, и все использованные атомы принадлежат своим доменам.

## 6. Юнит-тестирование (`tests.pl`)

Для автоматической проверки корректности базы знаний и предикатов был написан набор из **25 юнит-тестов** (с использованием библиотеки `plunit`).

Тесты покрывают:
- Отсутствие конфликтов (групп, преподавателей, аудиторий).
- Целостность доменов (нет неизвестных атомов).
- Количественные метрики (≥5 групп, ≥10 предметов и т.д.).
- Позитивные и негативные сценарии для `свободна_аудитория/3`.
- Корректность подсчёта `нагрузка_группы/2` (включая проверку, что сумма нагрузок всех групп равна общему числу фактов занятий).
- Краевые случаи для `окна_в_расписании/2` (отсутствие окон при одной паре, при двух парах подряд, наличие окна при разрыве).
- Smoke-тест на успешное выполнение всех 11 запросов без исключений.

Все 25 тестов успешно проходят (`passed`).
